# Руководство по внесению изменений (Contributing Guide)

Спасибо за ваш интерес к проекту! Этот документ содержит рекомендации по работе с репозиторием.

## Работа с GitHub Wiki

### Редактирование через веб-интерфейс

Самый простой способ редактирования Wiki:

1. Перейдите во вкладку [Wiki](../../wiki)
2. Выберите страницу для редактирования
3. Нажмите кнопку **"Edit"**
4. Внесите изменения в Markdown-разметке
5. Добавьте описание изменений в поле **"Edit message"**
6. Нажмите **"Save Page"**

### Работа с Wiki через Git

Для локального редактирования Wiki выполните следующие шаги:

#### Клонирование Wiki репозитория

```bash
# Клонируйте Wiki как отдельный репозиторий
git clone https://github.com/sl4sh73r/Software-Engineering-.wiki.git

# Перейдите в директорию Wiki
cd Software-Engineering-.wiki
```

#### Внесение изменений

```bash
# Создайте или отредактируйте .md файлы
# Например: nano Home.md

# Проверьте статус изменений
git status

# Добавьте измененные файлы
git add .

# Создайте коммит (см. рекомендации ниже)
git commit -m "Описание изменений"

# Отправьте изменения
git push origin master
```

#### Создание новой страницы в Wiki

1. Создайте новый файл с расширением `.md`
2. Название файла станет названием страницы (используйте CamelCase или дефисы)
3. Добавьте содержимое в формате Markdown
4. Добавьте ссылку на новую страницу в `_Sidebar.md` для навигации

Пример:
```bash
# Создайте новую страницу
echo "# Моя новая страница" > My-New-Page.md

# Добавьте и закоммитьте
git add My-New-Page.md
git commit -m "docs: добавлена страница My New Page"
git push origin master
```

## Рекомендации по написанию commit messages

Следуйте общепринятым соглашениям для написания понятных и информативных коммит-сообщений:

### Формат коммита

```
<type>: <краткое описание>

[опционально: подробное описание]

[опционально: футер с дополнительной информацией]
```

### Типы коммитов

- **feat**: новая функциональность
- **fix**: исправление ошибки
- **docs**: изменения в документации
- **style**: форматирование, отсутствующие точки с запятой и т.д. (без изменения кода)
- **refactor**: рефакторинг кода без изменения функциональности
- **test**: добавление или изменение тестов
- **chore**: обновление задач сборки, настроек и т.д. (без изменения кода)
- **ci**: изменения в CI/CD конфигурации

### Примеры хороших коммитов

```bash
# Хорошо
git commit -m "docs: добавлена документация по настройке окружения"
git commit -m "feat: добавлен GitHub Actions workflow для CI"
git commit -m "fix: исправлена опечатка в README.md"

# Плохо
git commit -m "обновление"
git commit -m "фикс"
git commit -m "правки документации"
```

### Дополнительные рекомендации

1. **Используйте повелительное наклонение**: "добавить" вместо "добавил" или "добавлено"
2. **Первая строка не более 50 символов**: краткое описание изменений
3. **Подробное описание при необходимости**: разделяйте первую строку и подробности пустой строкой
4. **Описывайте "что" и "почему"**, а не "как"

## Работа с Issues

При создании Issue:

1. Используйте понятный заголовок
2. Подробно опишите проблему или предложение
3. Добавьте соответствующие labels
4. Если возможно, предложите решение

## Работа с Pull Requests

При создании Pull Request:

1. Убедитесь, что ваш код соответствует стилю проекта
2. Опишите, какие изменения вы внесли
3. Укажите номер связанного Issue, если он есть
4. Дождитесь прохождения CI/CD проверок

## CI/CD и автоматизация

В процессе изучения курса "Программная инженерия" сюда будут добавлены:

- Правила написания и тестирования GitHub Actions workflows
- Требования к автоматическим тестам
- Процесс code review и требования к качеству кода
- Стратегии ветвления (branching strategies)
- Процесс релизов и версионирования
